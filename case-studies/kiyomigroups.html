The Problem

Every product in this portfolio, StreetTasker, AndreStylez, CV by Kiyomi, Swappy, Portfolia and TailorCV, was proof of real capability, but none of it lived anywhere together. Potential clients had no single place to see the range of work, understand how a project actually runs, or start a conversation. At the same time, a set of paid tutorials (Vibe Coding, UI/UX design, Illustrator, AI Mastery) had no storefront at all.

The studio needed one site to do two jobs at once, win client trust as a real software partner, and sell courses directly, without either function feeling bolted onto the other.

My Role

As the founder of the studio, I was the product owner, designer and developer on my own company's site, the one project in this portfolio where I was simultaneously the client and the builder.

Research

Before structuring the site, I looked at how founders actually evaluate a software studio before reaching out, and where similar agency sites lose people.

  • Process transparency builds trust — founders vetting a studio wanted to know how an engagement actually runs before booking a call, not just see a list of past work.
  • Portfolio needs proof, not claims — generic "we build great software" copy converts far worse than pointing directly at real, live, named products.
  • Two different visitor intents — some visitors wanted to hire the studio, others wanted to learn from it through the tutorials. Mixing those journeys on one page created confusion in early drafts.
  • Low friction first step — a single, low-commitment CTA (a free 20 minute discovery call) consistently outperformed asking visitors to fill out a long project brief immediately.

Design

Wireframes

I split the sitemap into six dedicated pages, Home, About Us, Services, Portfolio, How We Work and Contact, plus a separate Tutorials hub, so each page could carry a single clear job instead of trying to cover everything on one long homepage.

[ Hero: value prop + dual CTA ]
[ Who We Help: founders / businesses / teams ]
[ Built by Kiyomi: live portfolio grid ]
[ About: studio philosophy ]
[ What We Do: 3 service pillars ]
[ How We Work: 4-stage process timeline ]
[ Why Kiyomi: trust indicators ]
[ Tutorials: course cards + pricing ]
[ Contact: discovery call + project form ]
Mockups

Visual design leans structured and confident rather than playful, consistent card systems reused across both the portfolio grid and the tutorial storefront so the two very different offerings (custom builds vs. courses) still feel like one coherent brand.

Design System

A cool indigo-blue accent against deep navy, with a light mode toggle built in from the start since a professional B2B audience skews toward expecting one.

Development

Frontend

A multi-page HTML/CSS/JS build with a shared navigation and theme system across every page, including a working light/dark mode toggle persisted across the session.

Backend

Lightweight form handling powers the contact and discovery call flows, capturing project details and routing them for follow-up within 24 hours, without needing a full CRM for version one.

Database

Portfolio and tutorial content is structured as reusable data objects so new case studies or new courses can be added as new entries rather than new hand-built pages.

Authentication

No account system on the public site, every conversion point (discovery call, contact form, enrollment) is designed to work without requiring a visitor to sign up first.

Booking & Contact Flow

Built the discovery call booking path and the "send project details" form as two distinct, parallel CTAs so visitors can choose the commitment level that matches where they are in deciding to work with the studio.

Deployment

Deployed live to the kiyomigroups.com domain as the studio's permanent brand home, connecting outward to every individual product's own case study.

Challenges

One brand, two different buyers

Founders looking to hire the studio and learners looking to buy a tutorial are fundamentally different visitors with different intents. I solved this by giving Tutorials its own clearly separated section and navigation entry, rather than interleaving course cards into the client-facing portfolio narrative.

Selling process, not just output

Early versions of the site only showed finished work, which read as impressive but not necessarily trustworthy to a founder who has been burned by agencies before. Adding the explicit "How We Work" four-stage timeline, with realistic time estimates per stage, directly addressed that trust gap.

Keeping the portfolio connected without duplicating content

Each portfolio card needed to link out to a full, standalone case study rather than repeating the whole story on the studio site itself. I resolved this by treating Kiyomi Group's portfolio section as a summary layer, short description, key tags, one link, deep case study detail lives on each product's own dedicated page.

Revenue Channels

Kiyomi Group is now the studio's permanent home online, functioning as both a client acquisition engine and a course storefront from one coherent brand. It's also the clearest single artifact in this whole portfolio of what I actually do day to day: not just shipping individual products, but designing and running the business infrastructure that presents and sells that work.

Lessons Learned

Building your own studio's site is a different discipline from building a client's. There's no external deadline forcing scope decisions, which means it's easy to keep polishing indefinitely. The lesson that stuck was to treat my own site like a real client project, with a defined launch date and a clear "version one" scope, rather than letting it become a permanent work in progress.
