Scent of a Ghost

The featured story on the AdunolaBooks homepage — a psychological horror serial complete with genre tags, synopsis and a direct "Start Reading" flow into chapter one. This is the actual live cover art and layout structure pulled from the platform.

Homepage hero → story detail → chapter reader is the core loop the entire platform is built around.

The Problem

As a writer, I had stories and poetry with nowhere real to live. Publishing on generic third party platforms meant someone else's branding, someone else's ads, and no ownership over how the reading experience actually felt. A basic blog wasn't going to work either, serialized fiction needs structure: chapters, genres, a reading flow that pulls people back week after week, not just a wall of posts.

The challenge was building a platform that felt like a real literary destination, closer to a streaming catalog than a blog, while keeping publishing simple enough that new chapters could go up weekly without a full rebuild every time.

My Role

I was the product owner, designer and developer on this build, and also the person actually publishing content into it once it launched. That dual role, builder and end user, shaped a lot of the practical decisions, since I was going to be the one adding chapters every week.

Research

I looked closely at how readers actually behave on serialized fiction platforms like Wattpad and Webtoon, and what breaks the experience on generic blogging tools.

  • Discovery matters more than search — readers browse by genre and mood far more than they search by title, so genre tagging needed to be visible on every story card, not buried on a detail page.
  • Cover art drives clicks — on every platform I studied, the cover image was doing more work than the synopsis text in getting someone to click into a story.
  • Momentum keeps readers — a clear "next chapter" path at the end of every chapter was consistently the difference between a one-time reader and a returning one.
  • Reading comfort ≠ browsing style — the typography that works for scanning a catalog is wrong for actually reading long form text, these needed to be treated as two different visual modes.

Design

Wireframes

I mapped the site around three connected content types, stories, chapters and poems, before designing a single screen, so navigation and URL structure would make sense as the library grew.

[ Homepage: featured story hero + synopsis + CTA ]
[ "More from the Library" — secondary story cards ]
[ Featured Stories grid — genre tags + chapter count + status ]
[ Latest Chapters list ]
[ Poetry archive — standalone short pieces ]
[ Newsletter subscribe block ]
[ Story detail → Chapter reader → "Continue Reading" ]
Mockups

The homepage and browse views got a bolder, editorial type scale designed for scanning, while the chapter reader switched to a calmer, higher-contrast reading layout with generous line height, built specifically for long form comfort rather than browsing.

Design System

A dark, cinematic palette closer to a streaming platform than a book blog, letting cover art stay the visual anchor on every card, with genre tags as small accent chips layered on top.

Development

Frontend

Built with HTML, CSS and JavaScript, with story and chapter data rendered dynamically from the backend rather than hard coded, so publishing a new chapter never requires touching the front end code.

Backend

Supabase powers the platform's data layer, handling story records, chapter content and cover image storage through its object storage and database together.

Database

Stories, chapters and poems are modeled as related tables (a story has many chapters, a story has genre tags), which lets the catalog scale without restructuring every time a new title is added.

Authentication

Readers can subscribe by email without an account, keeping friction low for casual readers. Publishing access is restricted on the backend so only I can push new chapters live.

Content Pipeline

Cover art and chapter text upload through Supabase storage, decoupling content publishing from code deploys entirely, new chapters go live without a redeploy.

Deployment

Deployed to the adunolabooks.com domain with the backend connected live, so the site updates in real time as new stories and chapters are published.

Challenges

Making a static-feeling site actually dynamic

The homepage needed to feel curated and editorial, not like an auto-generated feed, while still being fully driven by live data. I solved this by designing a "featured story" slot that pulls from the database but is styled and structured like a hand-picked hero, rather than a generic latest-post block.

Two typography systems in one site

Getting the browsing type scale bold and scannable while keeping the chapter reader calm and readable meant maintaining two distinct typographic modes without the site feeling like two different products. I resolved this with a shared type family but different scale, weight and line-height rules scoped specifically to the reader view.

Publishing without a CMS dashboard

Building a full custom admin panel wasn't worth the time for a single-author platform. Instead I built a lightweight publishing flow directly against Supabase, fast enough for me to push a new chapter in minutes without needing a polished internal tool nobody else would ever use.

AdunolaBooks is live with an active, growing library and a genuine reader base returning for new chapters. The platform proved out something I wasn't fully sure of going in, that a fast, dynamic publishing system built by one person could hold up under weekly content updates without breaking or requiring constant developer attention.

Lessons Learned

The biggest lesson was separating content from code early. Early prototypes had story data hard coded directly into the pages, which worked for a demo but would have made weekly publishing painful. Moving story and chapter data into Supabase before launch, rather than after, saved enormous time later and is a decision I now make by default on any content-heavy build.
